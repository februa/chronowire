## 2. 依存方針

core必須依存は最小化する。

推奨:

- Python 3.11以上
- 標準ライブラリ中心
- `typing_extensions`は必要な場合のみ

任意依存:

- NumPy
- Graphviz
- Pydantic
- Rich
- pytest
- Hypothesis

Chronowire自体は任意のPython値を流せる設計とし、NumPyをcore必須にしない案が望ましい。

完了例:

```python
base = cw.Flow([1, 2, 3], config)
a = base.map(lambda x: x + 1)
b = base.map(lambda x: x * 2)

run_result = cw.compile([
    cw.output(a, collector=cw.Bounded(max_items=3)),
    cw.output(b, collector=cw.Bounded(max_items=3)),
]).run()
ra, rb = run_result.outputs
```

期待:

```text
[item.value for item in ra.emissions] == [2, 3, 4]
[item.value for item in rb.emissions] == [2, 4, 6]
```

共通Sourceは一度だけ読み出す。

## 8. Phase 5: 高速Backend

実装:

- 契約由来のgolden traceとPython参照Implementation
- profileで選んだhot loopのCython実装（必須の中間段階ではない）
- Cython/C++ DSP libraryをChronowire契約へ適配するOperation wrapper
- version付きC++ Operation ABIとnative buffer
- Python control planeから生成したPortablePlanIRを自立運用するC++ data plane
- Python Stage境界を含むmixed Planと、Python overheadをhot pathから除く完全native Plan
- serialized Plan IRと別processでの明示rebind

### 8.1 段階的最適化の実装gate

すべてのDSPをPythonからCython、C++の順に書き換えない。次のgateを独立に通す。

1. **意味論gate**: OperationSpec、PortablePlanIR、論理時間/status/Diagnostic契約から、
   値、interval、sequence、gap/reset、0/1/複数Emissionのgolden traceと不変条件を固定する。
   Python Implementation + PythonExecutorはそれを実行する参照実装として独立に検査する。
2. **局所最適化gate**: profileで支配するOperationだけをCythonまたはC++で実装する。
   Cythonはtyped buffer、`nogil`、C ABIの検証に有用だが、C++ firstでは省略できる。
3. **wrapper gate**: 外部DSPのアルゴリズム本体はChronowire型に依存させず、Operation wrapperで
   schema、Config packing、lifecycle、ownership、status/Diagnostic変換を試験する。
4. **Backend gate**: 同一OperationSpecをC++ Backend + PythonExecutorで実行し、native Operation単体を
   契約由来のgolden traceと照合する。この段階ではSchedulerはPythonのままでよい。
5. **data-plane gate**: C++ Backend + CppExecutorでSource ingressからcollector保持選択までをC++が所有し、
   run中のGIL解放、native Stage Python dispatch 0、NoCollect/native Sinkのoutput copy 0 byteを計測する。
6. **mixed-stage gate**: Python callable/Extensionを含むPlanでは境界をIRに記録し、batch変換、GIL取得、
   Python object復元数をprofilerで計数する。暗黙fallbackは行わない。

gate 2で目標性能を満たす場合はOperation本体をC++へ再実装する必要はない。一方、
PythonのSource materialization、Scheduler、buffer、collector復元が支配する場合はOperation単体の書き換えではなく、
gate 5のPlan運用全体を実装対象とする。

### 8.2 framework改修gate

新しいDSP Operationまたは既存ABI上のImplementation追加は、DSP package側の
OperationSpec、Implementation、native module manifest entry、conformance testで完結させる。
Chronowire本体のPythonBackend、NativeModuleBackend、PythonExecutor、CppExecutorは原則改修しない。

framework改修が必要なのは、新dtype、shape/Emission/output model、Config packing、workspace、
flush/checkpoint、ABI/process model、Implementation選択policy、またはScheduler/buffer/時間/status/lifecycleの
ExecutionPlan契約を拡張する場合である。後者は独立したPython/Cpp Executorの両実装に
変更が必要になり得る。全Executorを同時に対応させる必要はなく、未対応のExecutorは
Node、Port、binding、違反契約を含む明示エラーを返す。暗黙fallbackは行わない。

### v0.2

- 包含、overlap、tolerance付き同期
- PlanSessionによる継続state
- multiple output Port
- StateFlow Source helperとRuntimeOptions
- PortablePlanIRの明示binding
- runtime profilerと長時間streaming評価

### v0.3

- `executor.py`のExecutor protocolとPythonExecutor
- sample/block CBFの基準traceと実行粒度測定
- PortablePlanIR schema 0.3のStage、value schema、実験的Kernel ABI
- SOURCE、RATE、FRAME、単一MAPのCython Executor
- PortablePlanIRとnative bufferのsemantic conformance
- Python callbackのStage境界

`0.3.0.dev0`では上記のうちExecutor protocol、PythonExecutor、基準CBF、schema 0.3と、
有限f64 Sourceからidentity MAPまでの最小Cython経路を実装した。SoA native item、共有整数tick、
status byte、Diagnostic provenanceの実験ABIも同経路で検証した。汎用metadata、gap resetを含む
native buffer、native CBF、Python callback境界の実配送は引き続きv0.3範囲とする。

CBFアルゴリズムは本体へ含めず、`chronowire_reference`にPython/Cython参照実装を置く。
同packageはBackend交換と混在Stageのconformance用であり、将来のDSP Kernel packageが
実装すべき責務境界を示す。Cython CBF KernelとPython callbackの混在実配送は実装済みで、
Cython Executorによる固定shape多channel RATE/FRAME/CBF batch実行も実装済みである。
Stage間に残るPython method call、copy、Source/tick packing、collector復元は
`benchmarks/native_executor_cpp_gate.py`で計測済みである。測定によりCppExecutorはowned C ABI、
prepacked native ingress、遅延Emission復元を含む必要があると確定した。gap/INVALID partition、
汎用的な複数native Kernel連結は引き続きv0.3範囲とする。

### v0.4

- version付きKernel ABI
- C++ ExecutorとPython binding
- owned native buffer、prepacked integer-tick ingress、native Sink/NoCollect
- Python/C++ Backend交換
- streaming CBFのC++ Stage実行

`0.4.0`ではgeneric C++ DAG runtime、Python binding、prepacked Source/tick ingress、version付き
identity/固定CBF ABI table、native NoCollect/Latest/Boundedを実装した。同じExecutionPlanを
`executor="python"`、`"cython"`、`"cpp"`で選択でき、C++ sessionはRATE、FRAME、複数native MAP、
fan-out、collector保持選択をGILなしで自立運用する。gap reset、INVALID/DEGRADED、metadata、Diagnostic
provenance、複数output、有限PlanSession、one-shot Extension Stage境界をExecutor間で一致させた。
登録済みidentity/固定CBF/MVDR resolverは過渡的実装とし、今後のOperation追加は外部C ABI
module tableの汎用dynamic bindingを使う。

v0.4の確定範囲は単一の有限f64 vector Source、`FRAME(pad_end=False)`、statelessな登録済みnative ABI、
既定RuntimeOptionsである。realtime push、複数Source/merge、mutable native workspace、継続Extension
sessionはv0.5以降とし、暗黙fallbackせず契約名付きエラーにする。

### v0.5

- OperationSpec/ImplementationSpecの公開設計を実装し、legacy Kernelをadapter化
- compile-time shape unification、Config subtree/leaf依存、OperationDescriptorを追加
- Python first/C++ firstとC++ Backend + PythonExecutorのconformanceを確立
- block粒度、Fusion、Batch、parallel Stageの個別評価
- Executor間の性能・意味論比較
- representative evaluationと再現可能なbenchmark
- realtime push Source、複数Source/merge、mutable native Kernel sessionのABI設計
- C++ PlanSessionのincremental cursorと継続Extension session
- **初期実装済み**: Operation wrapper用ABI include APIと、Chronowire非依存のDSP本体の動的bind
- **Extension境界まで初期実装済み**: control-plane/data-plane profiler countersと完全native hot path gate。
  plain callableを含むmixed native Stageは未実装

Operation実装は[14_Operation設計.md](14_Operation設計.md)の段階順に進める。v0.4のKernel/ABIを一度に
削除せず、plain callableとPortablePlanIR schema 0.3をlegacy adapter/readerで維持する。
