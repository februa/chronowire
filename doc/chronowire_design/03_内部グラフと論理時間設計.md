# 内部グラフと論理時間設計

## 1. Graphの責務

Logical Graphは構築時の論理構造を保持する。

- Node登録
- Port登録
- Edge登録
- Source登録
- FlowとPortの対応
- Graph整合性維持
- GraphInfo生成

Graphは実行時バッファやScheduler状態を保持しない。

## 2. ID設計

内部識別子は整数IDを基本とする。

```python
NodeId = int
EdgeId = int
PortId = int
SourceId = int
GraphId = int
```

利点:

- C++へ移植しやすい
- 配列indexとして扱える
- serializeしやすい
- 比較・ハッシュが単純
- Graph最適化で扱いやすい

## 3. Node

```python
@dataclass(frozen=True)
class NodeSpec:
    id: NodeId
    kind: NodeKind
    input_ports: tuple[PortId, ...]
    output_ports: tuple[PortId, ...]
    operation_spec: OperationSpec | None
    attributes: Mapping[str, Any]
```

v0.4 Graph実装の`kernel_spec` fieldはOperation移行時に`operation_spec`へ正規化する。公開Operationの
入力名、shape、時間、status規則は[14_Operation設計.md](14_Operation設計.md)を正本とする。

Logical GraphのNodeKind候補:

```text
SOURCE
MAP
FRAME
RATE
TRIGGER
MERGE
STATE_READ
ADAPTER
```

`FUSED`や`BATCHED`はCompiled Graph側で生成する。

## 4. Port

```python
@dataclass(frozen=True)
class PortSpec:
    id: PortId
    node_id: NodeId
    direction: PortDirection
    index: int
    type_hint: Any | None
    time_semantics: TimeSemantics
```

Flowはoutput Portだけを指す。

## 5. Edge

```python
@dataclass(frozen=True)
class EdgeSpec:
    id: EdgeId
    source_port: PortId
    target_port: PortId
    semantics: EdgeSemantics
```

候補:

```text
STREAM
SYNCHRONOUS
STATE_LATEST
CONTROL
```

初期版はSTREAMとSYNCHRONOUSを統合してもよい。

## 6. 分岐

同一output Portに複数Edgeを接続できる。

```text
port 10
 ├ edge 20 -> node 5 input 0
 ├ edge 21 -> node 6 input 0
 └ edge 22 -> node 7 input 0
```

分岐Nodeは不要である。

## 7. 合流

複数input Portを持つNodeとして表す。

```python
result = signal.map(
    combine,
    reference=reference,
)
```

内部:

```text
signal.port    -> combine.input[0]
reference.port -> combine.input[1]
```

## 8. Graph上の終端

Logical Graph自体に固定終端はない。

```text
終端 = compile時に指定された観測Port
```

同じGraphから複数Planを生成できる。

```python
plan_a = cw.compile([beam])
plan_b = cw.compile([beam, covariance])
plan_c = cw.compile([preprocessed])
```

## 9. required Node集合

出力Port集合を`O`とすると、Planへ含めるNodeは次である。

```text
RequiredNodes(O)
  = Oを生成するNode
    ∪ その全祖先Node
```

未指定Flowの後続Nodeは含まない。

## 10. Graph不変条件

- Node IDは一意
- Port IDは一意
- Edge IDは一意
- Edge sourceはoutput Port
- Edge targetはinput Port
- Flowは存在するoutput Portを参照
- Graphはappend-onlyを基本とする
- 初期版では循環禁止
- 異なるGraph間のEdgeは禁止
- input Portの接続数はNode定義に従う

## 11. 循環と状態

初期版はDAGに限定する。

状態フィードバックはGraph cycleではなく、StateFlowまたはState Nodeで表す。

将来cycle対応を追加する場合も、ゼロ遅延cycleは禁止し、必ず明示的delayまたはstate境界を要求する。

# 論理時間

## 12. 目的

ChronowireはPython呼び出し順ではなく、値が表す時間区間を基準に同期する。

これにより次を保証する。

- 経路長が異なっても時間軸がずれない
- 分岐後の合流で対応区間を選べる
- frame境界を明示できる
- 複数rateを扱える
- Sourceへ必要区間を要求できる

## 13. 時間表現

浮動小数だけで境界を管理することは避ける。

推奨案:

```python
@dataclass(frozen=True, order=True)
class LogicalTime:
    ticks: int
    timebase_num: int
    timebase_den: int
```

または内部で`Fraction`相当を使用する。

C++移植性を考えると、整数ticksと有理timebaseが望ましい。

## 14. LogicalInterval

```python
@dataclass(frozen=True)
class LogicalInterval:
    start: LogicalTime
    end: LogicalTime
```

半開区間`[start, end)`を採用する。

## 15. StreamItem

```python
@dataclass(frozen=True)
class StreamItem(Generic[T]):
    value: T
    interval: LogicalInterval
    sequence: int
    status: EmissionStatus
    diagnostics: tuple[Diagnostic, ...]
    metadata: Mapping[str, Any]
```

値の型、shape、dtypeはChronowireが限定しない。

公開APIではStreamItemの読み取り専用表現を`Emission`と呼ぶ。`status`は`OK`、`DEGRADED`、`INVALID`のいずれかとする。statusはEdge、buffer、collector、Extensionを通過しても失われない。

- `OK`: 契約どおりの通常結果
- `DEGRADED`: fallbackや不十分な積分など、制約付きで観測可能な結果
- `INVALID`: 数値は利用できないが、区間と失敗情報を後段またはExtensionへ伝えられる結果

Operationは入力statusを確認し、受理、伝播、fallback生成のいずれを行うかをOperationSpecに記載する。

## 16. 同期規則

複数入力Nodeは論理区間が条件を満たしたとき起動する。

初期基本規則:

```text
完全同期:
    全入力intervalが一致

latest state:
    基準interval.start以前の最新状態を利用
```

v0.1は完全同期とlatest stateだけを実行対象とする。compile時に、既知のrate、offset、frame size/hop、timebase、interval duration、OperationSpecの時間変換宣言から完全一致を検証する。一般の経路で不一致を証明できる場合または情報不足で一致を証明できない場合は`POSSIBLE_INTERVAL_MISMATCH` warningを生成する。warningはcompileを停止しない。

ただしRATEを含む経路はframe境界をruntimeへ先送りしない。RATEを含む完全同期入力のduration/period不一致、未知time transformからRATEを介さないFRAME、完成済みFRAMEに対するRATEはcompile errorとする。利用者は外部resampling Operationを`time="explicit"`として旧格子を終了し、`rate(...).frame(...)`を明示して新格子を確定する。上流pull一回あたりのRATE出力件数が有理比により変動しても、RATEのoffsetとperiodがsession内で連続し、下流FRAMEがその時間格子上にある限りはframe境界不安定とはみなさない。

周期scheduleの`offset`は、論理時間0から発火時刻をどれだけずらすかを表し、発火時刻を`offset + n * period`で定める。これはDSP信号の位相ではない。信号位相はOperationが扱うdata側の概念とし、time descriptorへ混在させない。現行`Flow.rate()`のoffsetは0固定で公開引数にせず、利用例と必要性が確立した時点で`rate(..., offset=...)`を追加する。その際はexact mergeのduration、period、offset一致をcompile時に証明する。

run時は完全一致するintervalだけでNodeを起動する。Schedulerは必要intervalを各入力へ逆伝播し、不足経路だけを進める。あるproducerが必要intervalを通過して今後生成できないと確定した場合は、無制限に先行queueを増やさず`STALLED_EXACT_MERGE`を記録してそのNodeへの需要を停止する。有限SourceのEOFで初めて不一致が確定する場合も、Node、Port、待機intervalをDiagnosticへ記録する。

制御不能なリアルタイム入力で欠落が発生した場合は、内部`GapMarker`によって欠落intervalを同期Nodeへ伝える。exact mergeは欠落したintervalを永久に待たず、対応する待機itemを理由付きで解放して共通frontierから再開する。buffer、frontier、gapの詳細契約は[11_Buffer_Scheduler_PortablePlanIR設計.md](11_Buffer_Scheduler_PortablePlanIR設計.md)に定める。

再開は、failed Portに記録されたgap intervalとrequired intervalが重なる場合だけ許可する。単なる`skip()`、EOF、rate/phase不整合はgapとして推測せず、従来どおり`STALLED_EXACT_MERGE`で停止する。

v0.2で追加した同期方式:

- 包含同期
- overlap同期
- tolerance付き同期
- 明示resampling boundary。数値resampling自体は外部Operationが担当する

## v0.2同期選択規則

- `SYNCHRONOUS`: 両端が完全一致するintervalだけを一対一で選ぶ
- `CONTAINS`: candidateがreference全体を包含する場合に選び、広いcandidateは再利用できる
- `OVERLAPS`: 半開intervalの正のintersectionがある場合に選ぶ
- `TOLERANCE`: start差とend差の両方が明示tolerance以下の場合に選ぶ
- `LATEST`: reference開始時刻以下で確定した最新値を選ぶ

reference inputはMAP input index 0で固定する。候補はSource順序を維持し、複数候補ではsequence最小を選ぶ。各flexible inputは`SYNC_SELECTION` buffer一件と共有PortBufferの静的capacityだけを保持し、暗黙resamplingや無制限look-aheadを行わない。生成不能時はEdgeに固定した`STALL`または`SKIP`を適用する。

## 17. frameの時間意味論

```python
frames = flow.frame(size=1024, hop=512)
```

frame Nodeは履歴を保持し、必要サンプル数が揃うと出力する。

出力は代表timestampだけでなく、元入力に対応する区間全体を持つ。

```text
Frame 0: [0, 1024)
Frame 1: [512, 1536)
```

## 18. latency

区別する。

- algorithmic latency
- scheduling latency
- wall-clock latency

初期版ではalgorithmic latencyをGraph/Plan情報として扱う。

経路のNode数が多くてもlogical interval自体は変化しない。OperationSpecが明示的delayを持つ場合だけ時間変換する。

## 19. EOFと末尾処理

frame末尾不足:

- `pad_end=False`: 出力しない
- `pad_end=True`: paddingして出力

有限SourceのEOF後は、下流bufferをdrainしてからrunを終了する。
