# Chronowire v0.1確定事項と将来検討

v0.1は以下の推奨設計で実装する。ここに記載したv0.1項目は未決事項ではない。実装・性能測定により前提が崩れた場合は、変更理由と互換性への影響を設計書へ記録する。v0.1の必須実装、非対象、release gate、およびv0.2の確定方針は[12_v0.1_v0.2リリース方針.md](12_v0.1_v0.2リリース方針.md)を正本とする。

## 1. Flow生成

主APIは`Flow(source, config)`とする。`Flow.from_source()`、`Flow.from_iterable()`は、実際にconstructorの曖昧性が問題になった場合だけ追加する。

## 2. 論理時間

内部表現はinteger ticksとrational timebaseとする。Python実装の性能を測定するが、浮動小数秒だけを境界表現には使わない。

## 3. Kernel出力

公開Flow APIは単一output Portに限定する。Python callableの戻り値は次のとおり。

```python
return value                 # 1 Emission
return cw.skip()             # 0 Emission
return cw.emit_many(values)  # 複数Emission
```

通常のlistとtupleは一つの値として扱う。複数output PortはGraph IRとKernelResultだけが対応し、公開APIはv0.2以降に検討する。

## 4. Emission status

- `OK`は通常どおり後段へ渡す
- `DEGRADED`はDiagnosticを保持して後段へ渡す
- `INVALID`は、Kernelが明示的に受理しない限りKernelを呼ばず、そのまま伝播する
- 安全に継続できない契約違反だけを例外にする

安全なfallback、不十分な積分、利用不能だが観測可能な結果を暗黙にdropしない。

## 5. Config

Configは不変なscopeとして扱う。属性path単位で親scopeから子scopeへ明示的にマージする。Python callableの参照pathは`map(..., config_paths=(...))`を主APIとし、省略時はscope全体への依存とする。

時変制御値はFlow、最新状態はStateFlow、Kernel内部状態はKernelStateで表し、Config経由では移動しない。

## 6. merge同期

v0.1は完全interval一致とlatest stateに限定する。包含、overlap、tolerance、resampling boundaryはv0.2以降とする。

compile時に既知のrate、phase、frame size/hop、timebase、interval duration、時間変換宣言から完全一致を検証する。不一致または情報不足を検出した場合はwarningを出すが、compileは停止しない。

run時は観測終端から必要intervalを逆伝播し、不足経路だけを進める。producerが必要intervalを通過して生成不能と確定した場合は、EOFまでqueueを増やさず`STALLED_EXACT_MERGE`を記録して該当Nodeへの需要を停止する。`maximum_skew`は完全一致mergeへ導入せず、将来のtolerance付き同期で検討する。

## 7. compile outputとcollector

duplicate outputは`DuplicateOutputError`とする。bare Flowは`NoCollect`として扱う。

- `Latest`: 最新Emissionだけ保持
- `Bounded`: 上限付きで保持
- `Sink`: 逐次外部処理へ渡す

`Bounded`のdefault overflow policyは`FAIL`とする。`DROP_OLDEST`と`DROP_NEWEST`は明示指定時だけ使用し、drop件数をRunResultとDiagnosticへ記録する。`BLOCK`はrun終了までconsumerを持たないin-memory collectorでは成立しないため禁止し、queue-backed SinkまたはExtensionだけで扱う。

## 8. ExecutionPlanとstate

`CompiledKernel`はcompile済みfactoryとし、`plan.run()`開始時に`create_session()`でrun-localな`CompiledKernelSession`を生成する。KernelStateはsessionだけが所有し、同じPlanの次回runへ持ち越さない。継続実行はv0.2以降の`PlanSession`で扱う。

外部スレッドやUI向けのStateFlow Source helperはv0.1に含めない。低レベルSource protocolを使用し、利用例を踏まえてv0.2でAPIを検討する。

## 9. rateとSource chunk

`rate()`は論理時間、発火周期、Source要求周期に限定し、数値resamplingはKernel packageの責務とする。v0.1はHOLD policyだけを実装し、入力interval内の各発火時刻へ同じ値を割り当てる。周期は`Fraction`で保持し、generated Sourceのrequest幅には下流で必要な最短rate周期を使う。

Source chunkはv0.1ではruntime内部defaultを使用する。Configには入れない。利用者による調整が必要になった場合は、独立した`RuntimeOptions`を追加する。

## 10. Source modeとrealtime overflow

`scene_renderer`などSchedulerが生成を停止できるSourceは`PULL_CONTROLLED`とし、未処理需要またはbuffer余裕がない場合は追加生成しない。オーディオdevice、sensor、network入力など停止できないSourceは`REALTIME_PUSH`とし、`max_items`を必須とするbounded ingress bufferで受ける。

`REALTIME_PUSH`の既定overflow policyは`DROP_OLDEST`とし、明示指定時だけ`DROP_NEWEST`を許可する。`BLOCK`はリアルタイムcallbackでは禁止する。dropは`INPUT_OVERRUN` Diagnostic、欠落interval、drop件数を持つ内部`GapMarker`として伝播し、次のEmissionを`DEGRADED`とする。FRAMEは欠落をまたいだ履歴を暗黙に構成せず、stateful Kernelの既定gap policyは`RESET`とする。

## 11. Graphとexport

Graphはappend-onlyとし、compile時のrequired Node抽出で未使用経路を除外する。

`flow.export()`はFlowが属するGraph全体を出力する。祖先部分だけのexportは将来`scope="ancestors"`として追加できる。

## 12. PortBufferとmutability

fan-outは一つの読み取り専用`PortBuffer`を共有し、consumer cursorで寿命を管理する。全consumerが通過したitemだけを解放する。Kernel入力・出力は論理的にimmutableとし、debug modeではNumPyのwriteable flagや入力変更を検査できるようにする。in-place処理はcompileが一意所有権を証明できる場合だけ許可する。

通常の計算Portは暗黙dropを禁止し、静的上限を証明できない場合は明示`max_items`またはbackpressureを必須とする。複数Emissionを生成するKernelには一回の実行あたりの`max_items`を宣言させる。

## 13. Backend fallback

DEFAULT modeはwarning付きでNode単位のPython fallbackを許す。STRICT modeはfallbackせずcompile errorとする。

## 14. PortablePlanIR

PortablePlanIRはPython objectを含まず、Node、Port、Edge、Buffer、Time、Source、Extension、Binding descriptorを安定IDで参照する。Buffer descriptorは分類、`max_items`、任意の`max_bytes`、consumer cursor、overflow、reclaim、read-only、ownershipを持つ。Time descriptorはtimebase、duration、period、phase、時間変換を正規化した整数分子・分母で保持する。詳細は[11_Buffer_Scheduler_PortablePlanIR設計.md](11_Buffer_Scheduler_PortablePlanIR設計.md)を正本とする。

Extension観測は`cw.observe(flow, extension_id=..., trigger=...)`でcompile時に固定する。`extension_id`は必須かつPlan内で一意とし、観測Portをrequired rootおよびFusion境界とする。実体は`ExecutionPlan.create_session(extension_bindings=...)`で注入し、missing、unknown、unused、種別、ABI不一致を明示例外にする。`EveryLogicalTime`はEmissionの半開論理intervalで境界所属を決め、wall-clockを使用しない。

## 15. v0.2確定方針

- `PlanSession`による継続実行とresource lifecycle
- 包含、overlap、tolerance付き同期
- 複数output Portの公開Flow API
- 外部制御値向けStateFlow Source helper
- chunk、watermark、実行budgetを持つ`RuntimeOptions`
- plain callableのEmission上限、time transform、gap policyを宣言する統一API
- PortablePlanIRの明示bindingとv0.1 schema読込み
- session、buffer、Source lag、drop、Kernel時間のprofiler

Graph clone、builder scope、Flow祖先部分だけのexportは利用例に基づきv0.2内で採否を判断し、必須機能には含めない。Native Executorはv0.2の必須範囲に含めず、v0.3以降で最小Cython Executorから開始する。

## 16. 公開前確認

`chronowire`のPyPI、GitHub、主要ecosystem、ドメイン、商標、類似製品を確認する。名称を変更しても設計を維持できるよう、コードと公開文書に旧`spflow`名を残さない。
