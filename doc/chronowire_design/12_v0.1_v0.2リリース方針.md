# v0.1、v0.2リリース方針

## 1. versionの役割

Chronowireは、機能数ではなく実行意味論の安定段階でversionを区切る。

- v0.1: Python Executorを意味論の基準実装として完成させるrelease
- v0.2: 同じ意味論を継続sessionと実運用streamingへ広げるrelease
- v0.3以降: semantic conformanceを保った最適化とNative Executorの実証

v0.1とv0.2でDSPアルゴリズムをChronowire本体へ追加しない。FFT、CBF、resamplingなどの数値処理はKernel packageの責務とする。

package versionとGit tagは一対一に対応させる。release gate前はPEP 440のdevelopment releaseを使用し、現在の`0.1.0.dev0`には`v0.1.0.dev0`を対応させる。GitHubとPyPIへの公開手順はrepository rootの[RELEASING.md](../../RELEASING.md)に定める。

## 2. 現在のbaseline

現在の実装は、v0.1のAPIと基本実行系を確認できるbaselineである。

- `Flow`、`Config`、append-only Graph
- `map()`、`frame()`、`rate()`
- exact merge、latest state
- 0/1/複数Emission
- `CompiledKernel.create_session()`によるrun-local session
- finite iterable Sourceとpull型generated Source
- `NoCollect`、`Latest`、`Bounded`、`Sink`
- Extensionと`Snapshot`
- `OK`、`DEGRADED`、`INVALID`、Diagnostic伝播
- JSON/DOT export
- streaming CBF example

このbaselineはv0.1 release完了を意味しない。現行のconsumer別queueとSource round-robin取得は、[11_Buffer_Scheduler_PortablePlanIR設計.md](11_Buffer_Scheduler_PortablePlanIR設計.md)の契約へ置き換える必要がある。

## 3. v0.1の目的

v0.1は、Flowで記述した処理をPython Executorで安全かつ説明可能に実行し、同じ意味論を将来のExecutorが再現できるPortablePlanIRまでを完成させる。

### 3.1 v0.1必須機能

#### Flow、Kernel、run

- 単一output PortのFlow API
- `map()`、`frame()`、`rate(HOLD)`
- exact mergeとlatest state
- `skip()`、単一Emission、`emit_many()`
- Kernel一回あたりのEmission `max_items`
- runごとに分離された`CompiledKernelSession`
- finite runと論理duration指定run
- streaming CBFをpull-controlled Sourceから終端まで実行

#### 論理時間とScheduler

- integer ticksとrational timebase
- duration、period、phase、time transformのdescriptor
- 観測終端からのdemand逆伝播
- producer frontierと不足経路だけを進めるready判定
- exact mergeの静的整合検証
- `STALLED_EXACT_MERGE`による生成不能経路の停止
- 単一threadで決定的なPython Scheduler

#### bufferとfan-out

- 読み取り専用の共有`PortBuffer`
- consumer cursorによるfan-out寿命管理
- `PORT_SHARED`、`EDGE_ADAPTER`、`FRAME_HISTORY`、`LATEST_STATE`、`REALTIME_INGRESS`
- 静的capacityまたは明示`max_items`によるbounded buffer
- 通常の計算Portでの暗黙drop禁止
- 一意所有権を証明できる場合だけin-placeを許可

#### Source

- `PULL_CONTROLLED` Sourceとbackpressure
- `REALTIME_PUSH` Sourceの低level protocol
- 必須`max_items`を持つrealtime ingress
- `DROP_OLDEST`既定、明示`DROP_NEWEST`、`BLOCK`禁止
- 非dropでcoalesce可能な`GapMarker`
- `INPUT_OVERRUN`、欠落interval、drop件数の伝播
- gap後のFRAME、RATE、merge、stateful Kernelの既定処理

v0.1は実deviceごとのaudio、sensor、network adapterを本体へ含めない。test用RealtimeSourceでSource protocolとoverflow意味論を固定する。

#### 結果、診断、観測

- `OK`、`DEGRADED`、`INVALID`の保持と伝播
- 安全なfallback、不十分な積分、観測可能な失敗の保存
- Extension、collector、consumerの配送順固定
- bounded collectorとdrop Diagnostic
- `RunResult.completed`、status件数、Diagnostic、drop件数
- Kernel、buffer、Source、mergeの契約違反を区別できる例外

#### PortablePlanIR

- Node、Port、Edge、Buffer、Time、Source、Binding descriptor
- 安定IDとschema version
- Python objectを含まないJSON serialization
- 有理数の整数分子・分母表現
- execution domain、binding slot、Kernel ABI ID
- buffer capacity、ownership、cursor、overflow、reclaim条件
- JSON schema validationとserialize/deserialize round-trip試験

v0.1では、deserializeしたPortablePlanIRへ別processのbindingを与えて実行する公開APIまでは提供しない。公開`ExecutionPlan`はprocess-localな`ExecutionBindings`を内包し、exportしたIRが意味論を完全に説明できるところまでをrelease範囲とする。

### 3.2 v0.1非対象

- `PlanSession`によるrunをまたぐ継続実行
- 包含、overlap、tolerance付き同期
- 暗黙または自動の数値resampling
- 複数output Portの公開Flow API
- 外部制御値向けの高level StateFlow Source helper
- Fusion、Batch変換、parallel Stage実行
- compile cacheと性能profiler
- Cython/C++ Executor
- Python callbackの暗黙native化
- Chronowire本体へのDSPアルゴリズム内包

### 3.3 v0.1 release gate

以下をすべて満たした場合だけv0.1を完成とする。

1. 公開APIとv0.1設計書が一致する
2. 公開class、function、methodへ日本語docstringがある
3. Python Executorが共有`PortBuffer`とdemand-driven Schedulerを使用する
4. exact merge待ちで制御可能なSourceを無制限に先行させない
5. realtime overflowとgap伝播を決定的なtestで再現する
6. 値、interval、sequence、status、Diagnostic、metadataのconformance traceを固定する
7. PortablePlanIRが全Node、Port、Edge、time、buffer契約を失わずround-tripする
8. streaming CBFが通常入力と安全な劣化入力の両方で完走する
9. fan-out二重実行、buffer解放、0/1/複数Emission、EOF、gap、Extension失敗を試験する
10. `uv run pytest`、Pyright、Ruff check、Ruff format checkが成功する

## 4. v0.2の目的

v0.2は、v0.1で固定した一回のExecutionPlan実行を、長時間継続するstreaming session、外部制御、明示的な時間関係へ拡張する。v0.1の値、interval、sequence、status、Diagnostic、buffer寿命を変更しない。

### 4.1 v0.2必須機能

#### PlanSession

- `start()`、`run_until()`、`flush()`、`close()`、`cancel()`を持つ継続session
- runをまたいで保持するstateと、resetするstateの明示
- RealtimeSource sessionの開始、停止、fail、drain
- cancel、Source fail、Extension fail後の再実行可能性
- session単位のresource ownershipとDiagnostic

#### 拡張同期

- interval包含同期
- interval overlap同期
- 明示tolerance付き同期
- 各同期方式のreference input、選択規則、tie-break、欠落policy
- 同期方式ごとのbuffer上限とfrontier解析

resampling boundaryはGraphへ明示できるようにするが、数値補間処理は引き続き外部Kernelが担当する。Schedulerが暗黙に値を補間しない。

#### 公開API拡張

- 複数output Portを持つKernelResultとFlow handle
- 外部制御値向けStateFlow Source helper
- Source chunk、watermark、実行budgetを指定する`RuntimeOptions`
- plain callableへ`max_items`、time transform、gap policyを付与する統一adapterまたはdecorator

#### portable bindingと観測

- v0.1 PortablePlanIRを読み込み、明示`ExecutionBindings`でbindするAPI
- schema互換性検査とv0.1 IRの読込み保証
- session、buffer occupancy、Source lag、drop、Kernel時間のprofiler
- traceとDiagnosticを使った長時間streaming評価

### 4.2 v0.2非対象

- Python意味論を変更する性能最適化
- 自動Fusionまたは自動Batch変換
- parallel Stageの既定有効化
- C++ Executorのproduction化
- DSPアルゴリズムまたはdevice固有adapterのcore組込み

### 4.3 v0.2 release gate

1. v0.1の全conformance testを変更せず通す
2. 継続sessionの開始、flush、close、cancel、失敗回復を試験する
3. 各同期方式のboundednessと決定的tie-breakを試験する
4. v0.1 PortablePlanIRを読込み、明示bindingで同じtraceを再現する
5. 長時間RealtimeSource試験でbuffer上限を超えない
6. profilerを無効化した場合に実行意味論が変わらない
7. pytest、Pyright、Ruffと全page文書リンク検査が成功する

## 5. Native Executorへ進む条件

Native Executorの実装はv0.2の必須範囲に含めない。v0.1でPython意味論とPortablePlanIRを固定し、v0.2で継続sessionと明示bindingを安定化した後、v0.3以降で最小Cython Executorから開始する。

順序は次とする。

1. Python Executorのconformance trace固定
2. block単位streaming CBFの基準測定
3. SOURCE、RATE、FRAME、単一MAPに限定したCython Executor
4. Python/Cython間の値、時間、status、Diagnostic、gap同値性検証
5. Kernel ABIとnative buffer契約の固定
6. C++ Executor
