# Chronowire 試験設計

## 1. 方針

Chronowireはフレームワークであるため、個別例だけでなく不変条件を試験する。

- Graph構築の正しさ
- 実行順序
- 分岐時の一回実行
- 複数出力
- 論理時間
- 同期
- buffer解放
- compile最適化前後の等価性
- backend間等価性

## 2. Flow試験

- `Flow(source, config)`でGraph生成
- mapごとにNodeが追加される
- Flow chainでConfig scope参照が継承される
- `with_config()`で分岐ごとにscopeを変更できる
- Config経由の時変データ移動を禁止する
- Flow/StateFlow引数がEdgeとして記録される
- Flowに実データを保持しない
- 同一Flowから複数mapでfan-outできる
- 異なるGraphのFlow合流を拒否する

## 3. compile試験

### 3.1 単一出力

```python
cw.compile([a])
```

### 3.2 複数出力

```python
cw.compile([a, b, c])
```

### 3.3 途中出力

```python
cw.compile([base])
```

### 3.4 途中と後段

```python
cw.compile([base, beam])
```

### 3.5 未指定経路除去

`diagnostic`を指定しない場合、ExecutionPlanに固有経路が含まれないこと。

### 3.6 共通祖先

`preprocess`が分岐ごとに二重実行されず、一回だけ実行されること。

### 3.7 出力順序

```python
run_result = cw.compile([
    cw.output(b, collector=cw.Latest()),
    cw.output(a, collector=cw.Latest()),
]).run()
r0, r1 = run_result.outputs
```

が`b`, `a`の順に対応すること。

## 4. 論理時間試験

- 連続itemのintervalが隙間なく連結
- frame intervalが正しい
- hopによる重複が正しい
- 分岐経路でintervalが維持される
- wall-clock処理時間差でlogical timeが変わらない
- 有理timebaseで丸め誤差が蓄積しない
- RATEの初回発火、周期、gap、終了境界
- RATE通過後もstatus、Diagnostic、metadataを保持
- FRAME EOF paddingのinterval、sequence、status、Diagnostic順序
- `rate(...).frame(...)`のduration/periodが有理格子上で整数frameになる
- `frame(...).rate(...)`とpreserve MAPを挟む同等経路がcompile errorになる
- 明示time transform後はRATEを介さないFRAMEを拒否し、`rate(...).frame(...)`を許可する

## 5. merge同期試験

- 同一intervalで発火
- 片側不足時に待機
- EOF時の不足診断
- rateを含まないframe/timebase不一致可能性をwarning
- RATEを含むduration/period不一致をcompile error
- warning後もcompileを継続
- 不一致intervalを暗黙にdropせずEOF診断へ記録
- rate、phase、duration、time transformが一致する経路を静的に証明
- exact merge待ちで揃っている入力を先行pullしない
- producerが必要intervalを通過したら`STALLED_EXACT_MERGE`を記録して需要を停止
- stalled mergeと独立した観測終端は継続

## 5.1 PortBuffer、fan-out、realtime試験

- fan-out先が同じ読み取り専用itemを参照する
- 遅いconsumerが通過するまでitemを解放しない
- 全consumer通過後にitemを解放する
- Extension callback終了だけではitem寿命を延長しない
- `PULL_CONTROLLED` Sourceをhigh-watermark到達時に追加要求しない
- frame分岐需要を共有祖先Portだけのcapacityへ逆伝播
- nested frame需要を積で計画し、無関係Portはproducer burst上限のままにする
- `emit_many(max_items)`を生成Portだけのcapacity下限にする
- `capacity_reasons`とhigh/low watermarkをPortablePlanIR round-tripで保持
- `FRAME_HISTORY`のsize上限、hop prefix解放、所有Node/inputをround-tripで保持
- `LATEST_STATE`が確定値一件だけを置換し、future pendingを`PORT_SHARED`へ残す
- MAPが`skip()`したintervalをproducer frontierで即座に生成不能判定
- `REALTIME_PUSH`の一時的な遅延を`max_items`内で吸収
- `DROP_OLDEST`後に`INPUT_OVERRUN`、欠落interval、drop件数を記録
- `DROP_NEWEST`で既存itemを保持し、後続GapMarkerを順序どおり配送
- 連続dropをcapacity外の一つのGapMarkerへcoalesce
- gap直後のEmissionを`DEGRADED`にし、FRAME履歴を破棄してKernel sessionを再生成
- gapと重なるexact merge intervalだけを`MERGE_INPUT_GAP`で解放し、後続intervalを再開
- gap根拠のない`skip()`は`STALLED_EXACT_MERGE`のままにする
- realtime callbackで`BLOCK`を拒否
- gapをまたいだ未完成FRAMEを暗黙に構成しない
- gap後にstateful Kernelの既定policyでsessionをreset
- Kernelの複数Emissionが宣言`max_items`を超えた場合、途中出力を公開せず失敗
- latest state参照

## 6. buffer試験

- fan-outで不要copyしない
- 全consumer消費後に解放
- frame履歴長が正しい
- bounded buffer overflow
- NoCollectが値を保持しない
- Latestが最新値だけを保持する
- Boundedが上限を超えない
- collectorのdrop/failure件数を診断できる
- Extension観測はcollectorの有無と独立する
- run終了時drain

- 属性pathによる継承とleaf上書き
- 親scopeの不変性

- to_dict
- Nodeごとの参照path記録
- schema違反と型衝突

## 8. Kernel試験

- 引数1個の関数
- config自動注入
- 0/1/複数Emission
- `DEGRADED` fallback出力
- `INVALID`伝播
- 例外ラップ
- state reset
- compileが一回だけ
- plan.runを複数回
- backend fallback

## 9. Extension試験

- `NoCollect`なPortも観測できる
- trigger条件以外では呼び出されない
- priority、登録順で決定的に呼び出される
- initialize/finalizeが一回だけ呼び出される
- Snapshotが`DEGRADED`/`INVALID`とDiagnosticを保存する
- bounded queueのdrop件数が記録される
- Extension例外の継続/停止policyが守られる
- 同一EmissionをExtension、collector、consumerの順に配送する
- collector failure対象EmissionもExtensionが先に観測できる

## 10. 最適化等価試験

同一Graphについて:

```text
optimization=NONE
optimization=DEFAULT
```

の出力が等価であること。

比較対象:

- Emission values/status/diagnostics
- logical intervals
- sequence
- output ordering
- 例外発生位置
- 劣化結果の発生位置

将来のExecutor追加時はPythonExecutorのtraceを正本とし、RATE、FRAME、Source request、配送順、status、Diagnostic、metadata、sequenceについて同じ比較を全Executorへ適用する。

v0.1の固定traceは`tests/conformance/v0_1_python_trace.json`を正本とする。golden更新は意味論変更として扱い、値だけを都合よく更新せず、設計変更理由とExecutor互換性への影響を同時にreviewする。

## 12. 性能試験

- fan-out数とmemory
- frame sizeとthroughput
- output collector有無
- Fusion効果
- Python/Cython/C++比較
- peak memory
