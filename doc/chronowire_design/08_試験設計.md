# Chronowire 試験設計

## 1. 方針

Chronowireはフレームワークであるため、個別例だけでなく不変条件を試験する。

- Graph構築の正しさ
- 実行順序
- 分岐時の一回実行
- 複数出力
- 論理時間
- 同期
- buffer解放
- compile最適化前後の等価性
- backend間等価性

## 2. Flow試験

- `Flow(source, config)`でGraph生成
- mapごとにNodeが追加される
- Flow chainでConfig scope参照が継承される
- `with_config()`で分岐ごとにscopeを変更できる
- Config経由の時変データ移動を禁止する
- Flow/StateFlow引数がEdgeとして記録される
- Flowに実データを保持しない
- 同一Flowから複数mapでfan-outできる
- 異なるGraphのFlow合流を拒否する

## 3. compile試験

### 3.1 単一出力

```python
cw.compile([a])
```

### 3.2 複数出力

```python
cw.compile([a, b, c])
```

### 3.3 途中出力

```python
cw.compile([base])
```

### 3.4 途中と後段

```python
cw.compile([base, beam])
```

### 3.5 未指定経路除去

`diagnostic`を指定しない場合、ExecutionPlanに固有経路が含まれないこと。

### 3.6 共通祖先

`preprocess`が分岐ごとに二重実行されず、一回だけ実行されること。

### 3.7 出力順序

```python
run_result = cw.compile([
    cw.output(b, collector=cw.Latest()),
    cw.output(a, collector=cw.Latest()),
]).run()
r0, r1 = run_result.outputs
```

が`b`, `a`の順に対応すること。

## 4. 論理時間試験

- 連続itemのintervalが隙間なく連結
- frame intervalが正しい
- hopによる重複が正しい
- 分岐経路でintervalが維持される
- wall-clock処理時間差でlogical timeが変わらない
- 有理timebaseで丸め誤差が蓄積しない
- RATEの初回発火、周期、gap、終了境界
- RATE通過後もstatus、Diagnostic、metadataを保持
- FRAME EOF paddingのinterval、sequence、status、Diagnostic順序
- `rate(...).frame(...)`のduration/periodが有理格子上で整数frameになる
- `frame(...).rate(...)`とpreserve MAPを挟む同等経路がcompile errorになる
- 明示time transform後はRATEを介さないFRAMEを拒否し、`rate(...).frame(...)`を許可する

## 5. merge同期試験

- 同一intervalで発火
- 片側不足時に待機
- EOF時の不足診断
- rateを含まないframe/timebase不一致可能性をwarning
- RATEを含むduration/period不一致をcompile error
- warning後もcompileを継続
- 不一致intervalを暗黙にdropせずEOF診断へ記録
- rate、offset、duration、time transformが一致する経路を静的に証明
- exact merge待ちで揃っている入力を先行pullしない
- producerが必要intervalを通過したら`STALLED_EXACT_MERGE`を記録して需要を停止
- stalled mergeと独立した観測終端は継続

## 5.1 PortBuffer、fan-out、realtime試験

- fan-out先が同じ読み取り専用itemを参照する
- 遅いconsumerが通過するまでitemを解放しない
- 全consumer通過後にitemを解放する
- Extension callback終了だけではitem寿命を延長しない
- `PULL_CONTROLLED` Sourceをhigh-watermark到達時に追加要求しない
- frame分岐需要を共有祖先Portだけのcapacityへ逆伝播
- nested frame需要を積で計画し、無関係Portはproducer burst上限のままにする
- `emit_many(max_items)`を生成Portだけのcapacity下限にする
- `capacity_reasons`とhigh/low watermarkをPortablePlanIR round-tripで保持
- `FRAME_HISTORY`のsize上限、hop prefix解放、所有Node/inputをround-tripで保持
- `LATEST_STATE`が確定値一件だけを置換し、future pendingを`PORT_SHARED`へ残す
- MAPが`skip()`したintervalをproducer frontierで即座に生成不能判定
- `REALTIME_PUSH`の一時的な遅延を`max_items`内で吸収
- `DROP_OLDEST`後に`INPUT_OVERRUN`、欠落interval、drop件数を記録
- `DROP_NEWEST`で既存itemを保持し、後続GapMarkerを順序どおり配送
- 連続dropをcapacity外の一つのGapMarkerへcoalesce
- gap直後のEmissionを`DEGRADED`にし、FRAME履歴を破棄してKernel sessionを再生成
- gapと重なるexact merge intervalだけを`MERGE_INPUT_GAP`で解放し、後続intervalを再開
- gap根拠のない`skip()`は`STALLED_EXACT_MERGE`のままにする
- realtime callbackで`BLOCK`を拒否
- gapをまたいだ未完成FRAMEを暗黙に構成しない
- gap後にstateful Kernelの既定policyでsessionをreset
- Kernelの複数Emissionが宣言`max_items`を超えた場合、途中出力を公開せず失敗
- latest state参照

## 6. buffer試験

- initialize/finalizeが一回だけ呼び出される

- 同一EmissionをExtension、collector、consumerの順に配送する

## 10. 最適化等価試験

同一Graphについて:

```text
optimization=NONE
optimization=DEFAULT
```

の出力が等価であること。

比較対象:

- Emission values/status/diagnostics
- logical intervals
- sequence
- output ordering
- 例外発生位置
- 劣化結果の発生位置

意味論の正本はOperationSpec、PortablePlanIR、論理時間/status/Diagnosticの設計契約とする。
その契約からRATE、FRAME、Source request、配送順、status、Diagnostic、metadata、sequenceの
golden traceとpropertyを固定する。PythonExecutorとCppExecutorはそれぞれ独立にgolden/propertyを満たし、
さらにExecutor間の相互比較も行う。PythonExecutorの実行結果のみを他Executorの正本としない。

v0.3ではsample粒度とblock粒度の固定CBFについて、値、interval、sequence、status、Diagnosticを同一traceで比較する。Kernel呼出し回数とScheduler stepがblock粒度で減ることを検査し、Kernel実行時間は測定するが環境依存の閾値をtestへ入れない。Executor選択を差し替えても同じ契約由来traceになること、schema 0.3のStage/value schema/Kernel ABIがround-tripすること、schema 0.2で三descriptorが欠落しても読めることを試験する。

最小Cython Executorでは、明示f64 SourceからRATE、FRAME、identity MAP、collectorまでを
Python ExecutorとRunResult単位で比較する。整数rateだけでなく有理数period、空入力、
overlapするframeを含める。未対応のPython Source/callbackと継続sessionが暗黙fallbackせず
拒否され、明示f64 Port schemaとidentity ABIがPortablePlanIRに記録されることも試験する。
DEGRADED/INVALID、Diagnosticの重複順序、非zero有理Source interval、status集計も同じ比較へ
含める。`INPUT_OVERRUN` gap resetは対応するまで明示拒否を試験する。

固定CBF Backend conformanceでは、Python CBF、Cython CBF、Python前後処理とCython CBFの
混在構成、Cython Executor + Cython CBFを同じ出力traceで比較する。Cython構成のKernel ABIが
version付きnative互換としてIRへ記録され、混在構成の前後callbackがそれぞれPython Stage境界に
なることを確認する。Cython Executor構成ではSource/RATE/FRAME/CBFの全Portが固定schemaと
native buffer descriptorを持ち、CBF出力shapeもcompile時に確定することを試験する。

v0.4 CppExecutorでは契約由来のgolden traceを直接の比較対象とし、複数native MAP、fan-out、複数output、
NoCollect/Latest/Bounded、gap reset、INVALID/DEGRADED、metadata、Diagnostic provenanceを比較する。
fan-outの共通祖先は一runで一度だけ評価されること、有限PlanSessionの境界が厳密に単調であること、
closeはdrainしcancelは直前snapshotを保持すること、one-shot Extensionがtriggerとpriorityに従い
Python Stage境界で配送されることを試験する。8192 sample、64 sample frame、32 sample hopの固定CBFを
長時間回帰例とし、保持上限と全255 frameの受信件数も検査する。

Executor capabilityは段階導入を認める。新capabilityをまず一方のExecutorへ実装する場合、
対応済みExecutorは契約由来のgolden/property testを通し、未対応Executorは暗黙fallbackせず
Node、Port、binding slot、違反契約を含む明示エラーを返すことを試験する。同一ABI上の
新規DSP Operation追加では、Chronowire本体を変更せずDSP packageのmanifest entryとconformance testだけで
PythonExecutor/CppExecutorの対応済み経路を実行できることを回帰試験する。

周期MVDR受入試験では、`FRAME → 累積共分散 → SAMPLE → 重み生成 → LATEST → 重み適用`をschema 0.4
Operation Planとしてcompileする。更新時刻だけに重みEmissionが存在し、中間frameが直前の確定重みを
参照すること、SAMPLE周期がframe周期の整数倍でなければ`stable_sample_boundary` compile errorになる
ことを検査する。低頻度更新branchのlookaheadを共有frame bufferの有限`max_items`へ反映し、主branchを
停止させない。最初の積分sample数がchannel数の二乗未満でも、共分散、重み、beamを捨てずに
`DEGRADED`と`INSUFFICIENT_INTEGRATION`を伝播する。PythonExecutorとCppExecutorの完全traceを比較し、
同一Planの二回実行で共分散状態が持ち越されないことも確認する。

v0.1の固定traceは歴史的に`tests/conformance/v0_1_python_trace.json`という名前だが、
PythonExecutor実装そのものではなく設計契約から承認されたgolden artifactと位置付ける。
golden更新は意味論変更として扱い、値だけを都合よく更新せず、設計変更理由と
Python/Cppを含むExecutor互換性への影響を同時にreviewする。

## 12. 性能試験

計測項目:

- 1 Nodeあたりoverhead
- map chain長とthroughput
- fan-out数とmemory
- frame sizeとthroughput
- output collector有無
- Fusion効果
- Python/Cython/C++比較
- peak memory
- Source materialization、logical-time packing、native Scheduler、Kernel、collector復元の区間別latency
- Python/native遷移数、Stage dispatch数、payload copy回数とABI境界copy byte
- GIL保持区間、Python object復元数、RunResult/collector保持件数

`benchmarks/native_executor_cpp_gate.py`はblock size別のp50/p95/p99、throughput、Python heap peak、
静的copy内訳をJSONへ出力する。性能値をpytestの閾値にせず、測定schema、固定格子、copy計数だけを
回帰試験で固定する。初回baselineとC++移行判断は
[13_CppExecutor移行測定.md](13_CppExecutor移行測定.md)へ記録する。

schema 0.2はCythonに加えて、CppExecutorのsession生成込み、構築済みsession、全frameをPythonへ
戻すBounded経路を分けて測定する。C++内部のScheduler、Kernel、collector選択時間、owned input
byte、output boundary byte、Python/native遷移数、Stage Python dispatch数も同じJSONへ記録する。

完全native Planの構造的回帰試験では、native Stage Python dispatchが0、run中のC++ loopが
GILを保持しないこと、`NoCollect`/native Sinkのoutput boundary copyが0 byteであることを固定する。
Python/native遷移数は入力Emission数を変えても増えないことを確認する。`Latest`/`Bounded`では
Python object復元数が入力件数ではなく保持件数以下になることを検査する。Python Stageを挿入した
mixed Planは、境界数、batch変換数、GIL取得回数がPlan descriptorとprofilerで一致することを試験する。
